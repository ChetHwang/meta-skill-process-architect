# Lifecycle Operations — Process Skills

> **Full operational guide for maintaining generated process Skills across their lifespan.**
> Phase 1 stub → Phase 3 complete.
>
> **Industry-agnostic**: Bump, TTL-check, and diff-plan apply identically whether the process is refund management, insurance claims, order fulfillment, or patient intake.

---

## Overview

Every process Skill generated by meta-skill-process-architect has a lifecycle. Processes change — tools migrate, roles rename, SLAs shift, regulations update. This document defines how to detect, plan, and execute those changes systematically.

**Three core operations**:
1. **Bump** — increment VERSION + CHANGELOG when a change is made
2. **TTL Check** — scan for expired `@valid_until` markers and flag stale content
3. **Diff-Plan** — compare two versions and output an impact-aware update plan

These operations are available both as CLI commands and as agent-driven workflows.

---

## 1. Version Bump

### 1.1 Decision Table

Use the following table to determine the correct bump type. This is the canonical decision table from plan §14.2.

| What Changed | Bump Type | Example |
|---|---|---|
| Regulatory / compliance requirement change | **MINOR** or **MAJOR** | New data privacy law requires consent step → MINOR; process fundamentally restructured → MAJOR |
| Tool / system migration | **MINOR** | CRM migrates from Salesforce to HubSpot, URLs change |
| Org / role rename | **MINOR** or **MAJOR** (`[BREAKING]`) | "客服专员" → "客户成功经理" → MINOR; role split into 2 new roles → MAJOR |
| Volume / scale change (≥ 10×) | **MINOR** | Ticket volume from 100/day → 1000/day, SLA renegotiated |
| Incident / hotfix | **PATCH** or **MINOR** | Critical bug fix → PATCH; new exception path added → MINOR |
| Optimization / improvement | **MINOR** | Streamlined step sequence, reduced step count |
| Process retired / deprecated | **MAJOR** + `RETIRED` | Process no longer in use |

### 1.2 VERSION File Format

Semantic versioning: `MAJOR.MINOR.PATCH`

```
1.2.0
```

No trailing newline variations. Exactly `X.Y.Z`.

### 1.3 CHANGELOG Format

```markdown
# Changelog — process-{name}

## [1.2.0] — 2026-Q2

### Changed
- S4: Replaced manual approval with automated check (tool:Salesforce → HubSpot)
- S7: Updated SLA from 4h to 2h per Q2 optimization target

### Added
- S4.E: Exception path for automated check failure (fallback to manual)

### Fixed
- S3: Corrected role assignment (was "客服", now "售后专员")

---

## [1.1.1] — 2026-Q1

### Fixed
- S5: Fixed typo in tool URL

---

## [1.1.0] — 2026-Q1

### Changed
- L1.8: Updated resolution SLA from 8h to 4h per regulatory requirement

---

## [1.0.0] — 2026-Q1

### Added
- Initial release
```

### 1.4 Bump Workflow

**Pre-bump checklist**:
1. Identify what changed (regulatory / tool / org / volume / incident / optimization)
2. Map change to affected layers (L1.4 steps, L1.3 roles, L1.8 SLA, etc.)
3. Determine bump type from decision table
4. Run `bump --dry-run` to preview
5. Execute `bump` with changelog message
6. Run `validate` to confirm no regressions

## 2. TTL (Time-to-Live) Management

### 2.1 TTL Syntax

Fields that are likely to change over time must carry a `@valid_until` annotation:

```
SLA: "avg 4 hours @valid_until=2027-Q2"
Contact: "Zhang San (zhangsan@example.com) @valid_until=2026-Q4"
Tool URL: "https://crm.example.com/refund @valid_until=2026-Q3"
Role: "退款专员 (refund-specialist) @valid_until=2027-Q1"
Volume: "~200 tickets/month @valid_until=2026-Q4"
```

**Syntax rules**:
- Format: `@valid_until=YYYY-Q[1-4]`
- Must appear on the same line as the value it annotates
- Must be in a comment context (does not affect rendering)
- YAML frontmatter: `# @valid_until=YYYY-Q#` on the same line

### 2.2 TTL Application Rules

Apply TTL to fields that have a **known or probable expiry**:

| Field Category | Apply TTL? | Reason |
|---|---|---|
| SLA numbers (hours, days) | ✅ Always | SLAs are renegotiated periodically |
| Tool/system URLs | ✅ Always | URLs change on migration |
| Contact names/emails | ✅ Always | Personnel rotate |
| Role names (if person-specific) | ✅ Always | Org changes |
| Volume estimates | ✅ Always | Scale changes over time |
| Regulatory citations | ✅ Always | Laws/regulations update |
| Step sequence (L1.4) | ⚠️ If volatile | Only if this process changes frequently |
| Process purpose (L1.0) | ❌ Rarely | Purpose changes slowly |
| RACI assignments | ⚠️ If role-based | If role names have TTL, RACI inherits indirectly |
| Decision criteria (L1.5) | ❌ Rarely | Decision logic changes slowly |

### 2.3 TTL Check Rules

Run `ttl-check` periodically (recommended: start of each quarter):

| Condition | Status | Severity |
|---|---|---|
| `@valid_until` ≤ current quarter | `[TTL EXPIRED]` | ⚠️ Warn — must review |
| `@valid_until` = next quarter | `[TTL EXPIRING]` | ℹ️ Info — plan review |
| No `@valid_until` on volatile field | `[TTL MISSING]` | ℹ️ Info — recommend adding |
| `@valid_until` > next quarter | ✅ Valid | — |

### 2.4 TTL Check Report Format

```
TTL Check Report — process-customer-refund
Generated: 2026-Q2

Summary:
  Total fields checked: 12
  Valid: 8
  Expiring next quarter: 2
  Expired: 2
  Missing TTL (recommended): 3

EXPIRED (2):
  - L1.8 resolution SLA: "avg 8 hours @valid_until=2027-Q2" → Current: 2028-Q1
  - L1.3 role: "张经理 (refund-manager) @valid_until=2027-Q1" → Current: 2028-Q1

EXPIRING (3):
  - L1.3 tool URL: "https://old-crm.company.com @valid_until=2027-Q3"
  - L1.8 P0 SLA: "2h @valid_until=2027-Q3"
  - L1.4 S7 executor: "质检专员 @valid_until=2027-Q3"

MISSING (2):
  - L1.3 role: "售后专员" — has contact name, recommend adding TTL
  - L1.8 volume: "~500 tickets/month" — no TTL
```

## 3. Diff-Aware Update (diff-plan)

### 3.1 Overview

When a process changes, `diff-plan` compares the current Skill against a previous version (or a reference baseline) and outputs:
1. What changed, mapped to which layers
2. Which layers must be regenerated
3. Suggested bump type
4. Conflict detection (if two sources disagree on the same field)

### 3.2 Input

| Input | Source | Required? |
|---|---|---|
| Current Skill directory | `--skill ./process-xxx/` | ✅ |
| Baseline version | `--since 1.0.0` (reads from git) or `--since <path>` | ✅ |
| Optional: new materials | `--new-materials ./updated-sop/` | Optional |

### 3.3 Diff Categories

| Category | Example | Affected Layers |
|---|---|---|
| **Regulatory** | New law requires consent step | L1.4 (new step), L1.5 (new decision), L1.8 (new SLA) |
| **Tool** | CRM migrates | L1.4 (updated URLs in steps), L1.7 (tool column in RACI) |
| **Org** | Role rename / split | L1.3 (roles), L1.7 (RACI), L3 (recipes regen for renamed role) |
| **Volume** | Ticket count 10× | L1.8 (SLA renegotiation), L1.4 (parallelization) |
| **Incident** | New exception path | L1.4 (new .E/.A step), L3 (Exception Playbook) |
| **Optimization** | Streamlined steps | L1.4 (step sequence), L1.6 (state machine) |
| **Retirement** | Process deprecated | L0 (status: RETIRED), L1.1 (trigger removed) |

### 3.4 Impact Assessment Output

```yaml
diff_plan:
  process: "process-customer-refund"
  baseline: "1.0.0"
  target: "1.1.0 (proposed)"
  
  changes:
    - category: regulatory
      description: "New data privacy law Article 23 requires consent verification"
      affected_layers: [L1.4, L1.5]
      suggested_bump: MINOR
      conflicts: []

    - category: tool
      description: "CRM URL changed (Salesforce → HubSpot)"
      affected_layers: [L1.4]
      suggested_bump: PATCH
      conflicts: []

  overall_bump: MINOR
  layers_to_regenerate: [L1.4, L1.5, L3]
  estimated_effort: "~2 hours"
```

### 3.5 Diff-Plan Workflow

1. **Load baseline**: Read VERSION from `--since` or parse git history
2. **Extract current L1**: Parse process-brief.md for L1.1-L1.9
3. **Compare**: Match L1.4 steps by ID, L1.3 roles by name, L1.8 SLA by field name
4. **Classify**: Apply decision table to categorize each change
5. **Detect conflicts**: If `--new-materials` provided, cross-check against current Skill
6. **Output**: YAML report + suggested CLI bump command

## 4. Lifecycle State Machine

A generated process Skill goes through these lifecycle states:

```
[draft] ──(首次发布)──→ [trial] or [active]
[trial] ──(试运行结束)──→ [active]
[active] ──(新版本发布)──→ [superseded]
[superseded] ──(流程废弃)──→ [retired]
[active] ──(流程废弃)──→ [retired]
```

| State | Meaning | Action |
|---|---|---|
| **Draft** | Skill being authored, not yet published | No version tracking; no temporal window |
| **Trial** | Published with grace period (errors corrected, not penalized) | Record `valid_from` + `trial_until`; run `ttl-check` quarterly |
| **Active** | Current and formally enforced | Run `ttl-check` quarterly; SLA penalties apply |
| **Superseded** | Replaced by newer version; kept for historical audit | Archive; used by `which-version --at` for retroactive lookups |
| **Retired** | Process no longer in use | Archive Skill, mark CHANGELOG, remove from active validate set |

## 5. Schedule: Periodic Lifecycle Tasks

| Frequency | Task | Owner | CLI Command |
|---|---|---|---|
| **On every change** | Bump version + update CHANGELOG | Skill maintainer | `bump` |
| **Quarterly** | TTL scan | Process owner or automation | `ttl-check` |
| **On major change** | Diff-aware impact assessment | Process owner + Skill maintainer | `diff-plan` |
| **Annually** | Full process audit (TTL + anti-pattern + drift) | Process owner | `validate --trace` + `ttl-check` |

## 7. Temporal Query (which-version)

### 7.1 Purpose

When auditing a past event, determine which process version was authoritative at that time. Example:

```
Q: 3月15日的退款处理错了，5月核查时应该用哪个标准？
A: v1.0.0 (valid_from=2026-01-01, valid_until=2026-03-31, phase=active)
   NOT v1.1.0 (valid_from=2026-04-01, phase=active)
```

### 7.2 How It Works

Each published version records `valid_from` and (when superseded) `valid_until`. `which-version` scans a Skill's CHANGELOG to find the version whose temporal window contains the query date:

```
which-version --skill ./process-customer-refund/ --at 2026-03-15
→ v1.0.0 (active, 2026-01-01 to 2026-03-31)

which-version --skill ./process-customer-refund/ --at 2026-04-10
→ v1.1.0 (trial, 2026-04-01 to —, trial until 2026-04-30)
```

### 7.3 Audit Behavior by Phase

| Phase at time of event | Audit standard |
|---|---|
| **active** | Strict: SLA breach → penalty |
| **trial** | Lenient: SLA breach → correction, no penalty |
| **superseded** | Use this version's standard to judge the historical event |
| **draft** | N/A — drafts are never the authoritative version |

### 7.4 CLI Usage

```bash
# Which version was active on a specific date?
process-architect which-version --skill ./process-customer-refund/ --at 2026-03-15

# List all versions with their temporal windows
process-architect which-version --skill ./process-customer-refund/ --list

# Check if a specific version was in trial on a date
process-architect which-version --skill ./process-customer-refund/ --at 2026-04-10 --phase
# → "v1.1.0: trial (trial until 2026-04-30)"
```

## 8. Quick Reference

### CLI Commands

```bash
# Bump version
process-architect bump --skill ./process-customer-refund/ --type minor "Added consent verification step"
process-architect bump --skill ./process-customer-refund/ --type minor "Added consent verification step" --dry-run

# TTL check
process-architect ttl-check --skill ./process-customer-refund/
process-architect ttl-check --skills ./_e2e-test/  # batch check all Skills

# Diff plan
process-architect diff-plan --skill ./process-customer-refund/ --since 1.0.0
process-architect diff-plan --skill ./process-customer-refund/ --since ./backup/v1.0/
process-architect diff-plan --skill ./process-customer-refund/ --since 1.0.0 --new-materials ./updated-sop/
```
